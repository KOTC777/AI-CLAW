## 一、需求拆解

### 1. 备忘录
- 基础 CRUD（增删改查）
- 文本记录，可能需要富文本
- 与"灵感笔记"的区别：备忘录偏轻量、快捷

### 2. 日程表
- 日历视图 + 事件管理
- **系统级集成**：调用 AlarmManager 设置精确闹钟
- 通知提醒（NotificationChannel、前台服务）
- 与其他模块联动：为任务打卡提供提醒调度

### 3. 任务打卡
- 设定周期性/一次性任务
- 日程表驱动提醒
- **强制打卡机制**：未打卡 → 持续/升级提醒
- **打卡证明**：拍照/录音/文件等，验证后才停止提醒
- 核心难点：强制提醒的可靠性（系统杀死进程后的恢复策略）

### 4. 密码本
- 本地加密存储账号密码等敏感信息
- **迁移能力**：导出加密文件 → 导入到新设备
- 加密方案：Android Keystore + AES-256 + SQLCipher
- 主密码保护（非生物识别时的 fallback）

### 5. 灵感/笔记记录
- 模板系统（用户选择模板）
- 多媒体输入：文字、涂鸦（Canvas）、录音、链接、网页快照
- **AI 自动归类**：调用 AI API（如 OpenAI / 本地模型）
- 标签系统 + 智能分类
- 网页存储：WebView 加载 → 离线保存

## 二、关键技术决策

### 决策 1：原生 vs 跨平台

| 维度 | 原生 (Kotlin) | Flutter | React Native |
|---|---|---|---|
| 系统闹钟/通知 | ✅ 直接调用 | ⚠️ 需 Platform Channel | ⚠️ 需 Native Module |
| 前台服务 | ✅ 原生支持 | ⚠️ 插件质量参差 | ⚠️ 插件质量参差 |
| Android Keystore | ✅ 直接调用 | ⚠️ 需桥接 | ⚠️ 需桥接 |
| 涂鸦 Canvas | ✅ 原生 Canvas API | ✅ 自有 Canvas | ⚠️ 性能一般 |
| 开发效率 | 中 | 高 | 中高 |
| 后期维护 | ✅ 社区大、文档全 | ✅ Google 支持 | ⚠️ Meta 维护力度下降 |

**初步建议：Kotlin 原生** —— 你的需求中有大量系统级交互（强制通知、闹钟、前台服务、Keystore），跨平台方案在这些场景需要写大量原生代码，得不偿失。

### 决策 2：架构模式

**推荐：模块化 + MVVM + Clean Architecture**

```
app/                          # 壳工程
├── core/                     # 共享核心层
│   ├── core-common/          # 工具类、扩展函数
│   ├── core-database/        # Room + SQLCipher
│   ├── core-network/         # 网络请求（AI API）
│   ├── core-security/        # 加密、Keystore
│   └── core-ui/              # 共享 UI 组件
├── feature/                  # 功能模块（各自独立）
│   ├── feature-memo/         # 备忘录
│   ├── feature-schedule/     # 日程表
│   ├── feature-checkin/      # 任务打卡
│   ├── feature-password/     # 密码本
│   └── feature-inspiration/  # 灵感笔记
```

每个 feature 模块可独立编译、测试、升级，通过 Navigation 组件连接。

### 决策 3：数据层

| 组件 | 选型 | 理由 |
|---|---|---|
| 数据库 | Room + SQLCipher | 官方 ORM + 全库加密 |
| KV 存储 | DataStore | 替代 SharedPreferences |
| 文件存储 | Scoped Storage + SAF | Android 10+ 合规 |
| 密钥管理 | Android Keystore | 硬件级安全 |

### 决策 4：提醒系统

```
AlarmManager (精确闹钟)
    ├── setExactAndAllowWhileIdle()  → 关键提醒
    ├── BroadcastReceiver             → 接收闹钟
    ├── WorkManager                   → 后台补偿任务
    └── ForegroundService             → 强制打卡持续提醒
```

### 决策 5：AI 集成

- 云端 API（OpenAI / Claude / 国产大模型）→ 灵感归类
- 使用 Retrofit + OkHttp 调用
- 异步处理：Coroutines + Flow
- 离线时缓存待处理队列

## 三、待确认问题

请逐条回答以下问题，将直接影响最终架构方案：

### Q1：目标用户与设备
- a) 仅供自己使用，还是计划发布到应用商店？
- b) 最低支持的 Android 版本？（建议 Android 8.0+，API 26+）
- c) 是否需要支持平板/折叠屏？

### Q2：数据同步
- a) 是否需要多设备同步？（如果纯本地，架构简单很多）
- b) 如果需要同步，倾向云服务（自建/第三方）还是 P2P？

### Q3：密码本细节
- a) 是否需要自动填充（Accessibility Service / Autofill Framework）？
- b) 生物识别（指纹/面部）解锁是否必须？
- c) 密码生成器是否需要？

### Q4：强制打卡的"强制"程度
- a) 仅仅是持续通知提醒，还是需要更强制的手段（如锁屏覆盖、全屏弹窗）？
- b) 打卡证明材料的类型：仅拍照？还是支持录音、视频、文件？
- c) 是否需要设置"打卡截止时间"，超时后标记为失败？

### Q5：AI 能力
- a) AI API 偏好？（OpenAI / Claude / 国产如通义千问、文心一言 / 本地小模型）
- b) AI 主要用于分类，还是也需要摘要、续写、翻译等？
- c) 是否允许离线时无 AI 功能（降级为手动分类）？

### Q6：灵感笔记的网页存储
- a) "输入网页自动存储"是指保存完整网页（HTML+资源），还是仅保存 URL + 摘要？
- b) 是否需要离线阅读完整网页？

### Q7：UI 偏好
- a) 是否需要 Material Design 3（Material You）动态主题？
- b) 是否有暗色模式需求？
- c) 是否需要自定义主题/配色？

### Q8：其他
- a) 是否需要 Widget（桌面小组件）？
- b) 是否需要 Wear OS（手表）支持？
- c) 后期计划增加的功能方向？（提前预留扩展点）
