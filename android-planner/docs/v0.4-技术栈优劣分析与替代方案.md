# v0.4 — 技术栈优劣分析与替代方案

> 日期：2026-07-05
> 目的：逐层审视已选技术栈的优劣，探索可替代方案，确保决策经得起推敲

---

## 一、总体架构层

### 当前选择：原生 Kotlin 模块化单体

| 维度 | 优势 | 劣势 |
|---|---|---|
| 系统能力 | AlarmManager/前台服务/Keystore 直接调用，零桥接成本 | 仅限 Android，无法复用到 iOS |
| 性能 | 原生编译，启动快，内存占用小 | - |
| 生态 | Google 官方全力推进，Jetpack 组件成熟 | Compose 部分组件仍在演进（如 LazyColumn 性能边界） |
| 开发效率 | 中等，Compose 提升了 UI 效率 | 比 Flutter 略低（热重载不如 Flutter 完善） |
| 模块化 | Gradle Module 天然支持 | 模块间通信需设计（Navigation、接口抽象） |

### 替代方案 A：Flutter + 原生插件

| 维度 | 评估 |
|---|---|
| 优势 | 热重载极快、一套代码双端、UI 一致性高、Dart 语言简单 |
| 劣势 | AlarmManager/前台服务/Keystore 全部需要 Platform Channel 桥接；强制打卡的前台服务、锁屏覆盖等场景需要写大量原生 Kotlin 代码；SQLCipher 没有官方 Flutter 插件，需自己维护桥接层 |
| 适用场景 | UI 为主、系统交互少的应用 |
| **结论** | ⚠️ 你的需求约 40% 是系统级交互（闹钟/前台服务/加密/相机/录音），Flutter 在这些场景反而需要写双份代码。**不推荐** |

### 替代方案 B：React Native + Native Modules

| 维度 | 评估 |
|---|---|
| 优势 | JS/TS 生态庞大、社区组件多、热重载 |
| 劣势 | 性能瓶颈（JS Bridge）、前台服务/AlarmManager 需 Native Module、Facebook 对 RN 维护力度波动、新架构（Fabric/TurboModules）仍在迁移中 |
| 适用场景 | 内容展示型、轻交互应用 |
| **结论** | ❌ **不推荐**。强制提醒、加密、多媒体录制等场景 RN 劣势明显 |

### 替代方案 C：Kotlin Multiplatform (KMP) + Compose Multiplatform

| 维度 | 评估 |
|---|---|
| 优势 | 共享业务逻辑层（网络/数据/加密），UI 层用 Compose Multiplatform 可跨端；Android 侧完全原生 |
| 劣势 | Compose Multiplatform for iOS 仍处于 Alpha（2024-2025 才 Beta）；目前主要用于 Android + Desktop；iOS 支持不成熟 |
| 适用场景 | 需要未来扩展到 iOS 但当前先做 Android |
| **结论** | 🔶 **值得考虑作为远期方向**。如果未来可能做 iOS，现在可以把 core 层用 KMP 写（纯 Kotlin），feature 层用 Compose Android。迁移成本低 |

### 替代方案 D：PWA / Web App

| 维度 | 评估 |
|---|---|
| 优势 | 跨平台、无需安装、开发快 |
| 劣势 | 无法调用 AlarmManager/前台服务（Service Worker 能力有限）、无法访问 Keystore、无法做强制弹窗覆盖、通知能力受限、离线能力弱 |
| **结论** | ❌ **完全不适用**。你的核心需求（强制提醒、硬件加密、全屏覆盖）Web 无法实现 |

### 架构层结论

```
原生 Kotlin 模块化单体  ✅ 最佳选择
KMP + Compose Multiplatform  🔶 远期升级路径（如需 iOS）
Flutter  ⚠️ 系统交互多时不推荐
React Native  ❌ 不推荐
PWA  ❌ 不适用
```

## 二、UI 框架层

### 当前选择：Jetpack Compose + Material 3

| 维度 | 优势 | 劣势 |
|---|---|---|
| 开发效率 | 声明式 UI，代码量比 XML 少 40-60% | 复杂列表（LazyColumn）嵌套时性能需调优 |
| 状态管理 | State + Flow 天然绑定，单向数据流 | 学习曲线（副作用 LaunchedEffect/DisposableEffect 理解成本） |
| 主题 | Material 3 原生支持动态取色 | 自定义主题深度定制时不如 XML 灵活（某些 View 属性 Compose 未暴露） |
| 动画 | Animation API 丰富 | 复杂手势+动画组合时调试困难 |
| 生态 | Google 主推，新组件优先 Compose | 部分旧库仍需 AndroidView 桥接（如 WebView、MapView） |
| 成熟度 | 2024 年已 production-ready | 偶有编译器 bug（Compose Compiler 与 Kotlin 版本强绑定） |

### 替代方案：传统 XML View 系统

| 维度 | 评估 |
|---|---|
| 优势 | 极度成熟、社区资源最多、所有第三方库原生支持、WebView/MapView 等直接使用 |
| 劣势 | 命令式 UI，代码量大；数据绑定需要 DataBinding/ViewBinding 额外配置；主题切换不如 Compose 灵活 |
| **结论** | 🔶 如果团队对 Compose 不熟悉，XML 仍是安全选择。但新项目 2026 年起步，Compose 是正确方向 |

### 混合方案：Compose 为主 + AndroidView 桥接

```
整体 UI: Compose
WebView: AndroidView { WebView(context) }    ← 网页快照需要
Canvas 涂鸦: Compose Canvas + 手势检测        ← 原生支持
CameraX 预览: AndroidView { PreviewView }     ← 拍照需要
```

**结论**：✅ **保持 Compose**，必要时用 AndroidView 桥接 WebView/Camera 等传统组件。这是当前最佳实践。

## 三、数据存储层

### 当前选择：Room + SQLCipher

| 维度 | 优势 | 劣势 |
|---|---|---|
| 安全性 | SQLCipher 全库加密，AES-256 | 性能有损耗（约 5-15%，取决于查询量） |
| 类型安全 | Room 编译期校验 SQL | 复杂查询仍需写 SQL |
| 迁移 | 自动 Migration + 手动 fallback | 跨版本迁移时需仔细测试 |
| 生态 | Google 官方，与 Hilt/Flow 深度集成 | SQLCipher 社区维护，非官方 |

### 替代方案 A：Room（不加密）+ 字段级加密

| 维度 | 评估 |
|---|---|
| 优势 | 性能更好（无全库加密开销）；只有敏感字段加密 |
| 劣势 | 需要为每个敏感字段手动加解密；元数据泄露风险（表名、行数、非敏感字段可被看到） |
| **结论** | 🔶 可行，但安全性低于全库加密。对于密码本场景，全库加密更安心 |

### 替代方案 B：Realm / ObjectBox

| 维度 | 评估 |
|---|---|
| 优势 | 对象数据库，无需 SQL；ObjectBox 性能极快；Realm 支持加密 |
| 劣势 | ObjectBox 加密需付费；Realm 已被 MongoDB 收购，移动端维护力度下降；两者与 Compose/Flow 集成不如 Room 原生；SQLCipher 是行业标准，审计充分 |
| **结论** | ⚠️ **不推荐**。Room + SQLCipher 组合更成熟、更可控 |

### 替代方案 C：纯 DataStore（Preferences + Proto）

| 维度 | 评估 |
|---|---|
| 优势 | 极简，适合 KV 数据 |
| 劣势 | 不适合关系型数据（密码条目与分组的关联、打卡记录与任务的关联）；Proto DataStore 学习成本高 |
| **结论** | ❌ **不适合**。你的数据模型是关系型的（任务↔记录、笔记↔附件、密码↔分组） |

### 替代方案 D：SQLite 直接操作（不用 Room）

| 维度 | 评估 |
|---|---|
| 优势 | 最灵活，无 ORM 开销 |
| 劣势 | 手写 SQL 容易出错；无编译期校验；无 Flow 自动观察；样板代码多 |
| **结论** | ❌ **不推荐**。Room 在 SQLite 之上提供了编译期安全 + Flow 集成，几乎没有理由直接写 SQLite |

### 存储层结论

```
Room + SQLCipher  ✅ 最佳选择（安全 + 类型安全 + 生态）
Room + 字段级加密  🔶 轻量化替代（性能优先时可考虑）
Realm / ObjectBox  ⚠️ 生态风险
纯 DataStore  ❌ 不适合关系型数据
裸 SQLite  ❌ 开发效率低
```

## 四、依赖注入层

### 当前选择：Hilt

| 维度 | 优势 | 劣势 |
|---|---|---|
| 安全性 | 编译期检查，缺失依赖直接编译失败 | 编译速度有影响（KSP 处理注解） |
| 集成 | 与 ViewModel/Navigation/Room 官方深度集成 | 学习曲线（Scope/Component 理解成本） |
| 模块化 | `@InstallIn` 天然支持多模块 | 模块间共享依赖需设计 |

### 替代方案：Koin

| 维度 | 评估 |
|---|---|
| 优势 | 纯 Kotlin DSL，无注解处理，编译快；运行时注入，上手简单；Compose 集成好（`koinViewModel()`） |
| 劣势 | 运行时检查（缺失依赖在运行时才崩溃）；大型项目中依赖图不如 Hilt 清晰 |
| 编译速度 | Koin 快 ~15-30%（无 KSP） |
| **结论** | 🔶 **Koin 是有力替代**。如果你更看重编译速度和简洁性，Koin 是好选择。如果你更看重编译期安全，Hilt 更好 |

### 对比表

| 维度 | Hilt | Koin |
|---|---|---|
| 编译期安全 | ✅ | ❌ 运行时 |
| 编译速度 | 中 | 快 |
| 学习曲线 | 中高 | 低 |
| 多模块支持 | ✅ 原生 | ✅ 手动 |
| 代码量 | 中（注解） | 少（DSL） |
| 社区趋势 | Google 推荐 | Kotlin 社区推荐 |

**结论**：✅ **保持 Hilt**，理由是多模块架构 + 编译期安全。如果后续觉得 Hilt 编译慢，可切换到 Koin（迁移成本低，约 1-2 天）。

## 五、网络层

### 当前选择：Retrofit + OkHttp + Moshi

| 维度 | 优势 | 劣势 |
|---|---|---|
| 成熟度 | 行业标准，90%+ Android 项目使用 | 样板代码较多（接口定义） |
| 生态 | 拦截器生态最丰富（日志/重试/缓存/鉴权） | - |
| Moshi | 比 Gson 快，Kotlin 友好 | 比 kotlinx.serialization 多一层 |

### 替代方案 A：Ktor Client

| 维度 | 评估 |
|---|---|
| 优势 | Kotlin 原生，协程内置；与 Ktor Server 共享模型；API 简洁 |
| 劣势 | 生态不如 Retrofit（拦截器少）；Android 社区使用率低；文档偏少 |
| **结论** | 🔶 可行，但 Retrofit 生态优势明显 |

### 替代方案 B：Kotlinx.Serialization 替代 Moshi

| 维度 | 评估 |
|---|---|
| 优势 | JetBrains 官方；编译期生成，无反射；多平台支持（KMP） |
| 劣势 | 与 Retrofit 集成需额外适配器；API 不如 Moshi 直觉 |
| **结论** | 🔶 如果未来走 KMP 路线，kotlinx.serialization 是更好的选择。当前纯 Android，Moshi 足够 |

### 网络层结论

```
Retrofit + OkHttp + Moshi  ✅ 最稳妥选择
Retrofit + OkHttp + kotlinx.serialization  🔶 KMP 友好
Ktor Client  🔶 可行但生态弱
```

## 六、加密方案层

### 当前选择：Argon2id + AES-256-GCM + Android Keystore

| 维度 | 优势 | 劣势 |
|---|---|---|
| Argon2id | 2015 年密码哈希竞赛冠军；抗 GPU/ASIC 攻击；内存硬 | Android 原生无库，需引入 BouncyCastle 或独立实现 |
| AES-256-GCM | 认证加密（AEAD），防篡改 | - |
| Android Keystore | 硬件级安全（TEE/SE），密钥不可导出 | 部分低端设备 Keystore 实现有 bug |

### 替代方案 A：PBKDF2 替代 Argon2

| 维度 | 评估 |
|---|---|
| 优势 | Android 原生支持（`SecretKeyFactory`）；无额外依赖 |
| 劣势 | 抗 GPU 能力弱于 Argon2（纯计算密集，无内存硬） |
| 安全性 | PBKDF2 600k 迭代 ≈ Argon2id 中等参数（但 Argon2 可调内存参数进一步拉大差距） |
| **结论** | 🔶 **PBKDF2 是更轻量的替代**。如果不想引入 BouncyCastle（增加 APK ~2MB），PBKDF2 完全够用。密码本场景不是高频破解场景 |

### 替代方案 B：AndroidX Security (EncryptedSharedPreferences / EncryptedFile)

| 维度 | 评估 |
|---|---|
| 优势 | Google 官方封装，API 极简；底层用 Keystore + AES-256-GCM |
| 劣势 | 只支持 KV 和文件，不支持数据库字段级加密；性能开销比 SQLCipher 全库加密小，但灵活性也低 |
| **结论** | 🔶 可用于存储 AI API Key 等少量 KV 数据，不适合密码本的结构化加密存储 |

### 加密方案对比

| 方案 | 安全性 | 性能 | 依赖大小 | 推荐场景 |
|---|---|---|---|---|
| Argon2id + AES-256-GCM | ⭐⭐⭐⭐⭐ | 中 | +2MB (BouncyCastle) | 高安全需求 |
| PBKDF2 + AES-256-GCM | ⭐⭐⭐⭐ | 中 | 0（原生） | 轻量化 |
| SQLCipher 全库加密 | ⭐⭐⭐⭐⭐ | 中低 | +3MB | 全库保护 |
| EncryptedSharedPreferences | ⭐⭐⭐⭐ | 快 | +0.5MB | KV 数据 |

### 推荐组合

```
轻量方案：PBKDF2 + AES-256-GCM + Android Keystore + Room（不加密）
         → 性能最好，APK 最小，安全性足够（自用场景）

安全方案：Argon2id + AES-256-GCM + Android Keystore + SQLCipher
         → 安全最高，APK +5MB，性能略降

折中方案：PBKDF2 + AES-256-GCM + Android Keystore + Room + 字段级加密
         → 平衡选择，只有密码本字段加密
```

## 七、架构模式层

### 当前选择：MVVM + Clean Architecture

| 维度 | 优势 | 劣势 |
|---|---|---|
| 分层清晰 | UI → ViewModel → Repository → DataSource，职责明确 | 层级多，简单功能也要写 3-4 个类 |
| 可测试 | 每层可独立单元测试 | 测试代码量大 |
| 可维护 | 大型项目长期维护友好 | 小项目过度设计 |

### 替代方案 A：MVI（Model-View-Intent）

| 维度 | 评估 |
|---|---|
| 优势 | 单一数据源（State）；状态可追溯（时间旅行调试）；与 Compose 天然契合 |
| 劣势 | 样板代码多（Intent/Reducer/Effect）；学习曲线高；简单场景过于复杂 |
| **结论** | 🔶 Compose + 单向数据流本质上已经是 MVI 的变体。可以吸收 MVI 思想但不必严格遵循完整 MVI 框架 |

### 替代方案 B：简化 MVVM（去掉 Clean Architecture 的 UseCase 层）

| 维度 | 评估 |
|---|---|
| 优势 | 减少一层，ViewModel 直接调 Repository；代码量减少 30% |
| 劣势 | 复杂业务逻辑会堆积在 ViewModel 中 |
| **结论** | ✅ **推荐采用**。自用项目不需要严格的 UseCase 层。ViewModel 直接调 Repository，复杂逻辑用 Repository 内部方法封装 |

### 架构模式对比

```
Clean MVVM (严格分层)  → 大型团队项目
简化 MVVM (无 UseCase)  → ✅ 中小型项目/自用（推荐）
MVI (单向数据流)  → 状态复杂的应用
无架构 (Activity 直接操作)  → ❌ 不推荐
```

## 八、提醒系统层

### 当前选择：AlarmManager + WorkManager + ForegroundService

| 维度 | 优势 | 劣势 |
|---|---|---|
| AlarmManager | 精确闹钟（`setExactAndAllowWhileIdle`），Doze 模式下也能唤醒 | Android 12+ 需要 `SCHEDULE_EXACT_ALARM` 权限（用户可拒绝） |
| WorkManager | 后台补偿，进程重启后自动恢复 | 不精确（系统可能延迟执行） |
| ForegroundService | 前台服务不会被系统杀死（显示通知） | Android 14+ 需要声明前台服务类型（`specialUse`） |

### 替代方案：JobScheduler / SyncAdapter

| 维度 | 评估 |
|---|---|
| JobScheduler | 系统级调度，但不精确；WorkManager 底层已封装它 |
| SyncAdapter | 已过时，Google 不推荐 |
| **结论** | ❌ 不如当前方案 |

### Android 版本兼容性风险

```
Android 12 (API 31): 需 SCHEDULE_EXACT_ALARM 权限
Android 13 (API 33): 需 POST_NOTIFICATIONS 权限（运行时申请）
Android 14 (API 34): 前台服务需声明类型
Android 15 (API 35): 闹钟权限进一步收紧

应对策略：
  - 降级方案：精确闹钟不可用时，退化为 WorkManager 不精确调度
  - 用户引导：权限被拒时引导用户到系统设置手动开启
  - 检测：启动时检查权限状态并提示
```

**结论**：✅ **当前方案最优**。没有更好的替代方案能同时满足精确唤醒 + 后台存活 + 多版本兼容。

## 九、网页存储层

### 当前选择：WebView 单 HTML 快照（内联资源）

| 维度 | 优势 | 劣势 |
|---|---|---|
| 简洁性 | 一个文件搞定，管理极简 | 文件可能很大（图片 Base64 膨胀 33%） |
| 可移植性 | 单文件可直接分享/迁移 | CSS/JS 内联可能破坏某些网页布局 |
| 实现复杂度 | 低 | 需处理跨域资源、懒加载图片、字体等 |

### 替代方案 A：HTTrack 风格完整镜像

| 维度 | 评估 |
|---|---|
| 优势 | 完整保存，包括外部 CSS/JS/字体/图片 |
| 劣势 | 文件多、目录结构复杂、管理困难 |
| **结论** | ❌ 你要求轻便，完整镜像太重 |

### 替代方案 B：仅保存 URL + 截图

| 维度 | 评估 |
|---|---|
| 优势 | 极小存储占用 |
| 劣势 | 网页失效后无法访问；不是"完整存储" |
| **结论** | ⚠️ 可作为备选模式（快速收藏），但不满足"完整网页存储"需求 |

### 替代方案 C：Mozilla Readability + 单 HTML

| 维度 | 评估 |
|---|---|
| 优势 | 提取正文内容，去除广告/导航，文件更小 |
| 劣势 | 失去原始页面样式；不适合所有网页（如交互式页面） |
| **结论** | 🔶 可作为"阅读模式"选项，与完整快照并存 |

### 网页存储结论

```
单 HTML 内联快照  ✅ 主模式（轻便、完整）
Readability 正文提取  🔶 可选附加模式
完整镜像  ❌ 太重
仅 URL + 截图  ❌ 不满足需求
```

## 十、总结：最终推荐 vs 替代方案

| 层 | 当前选择 | 最佳替代 | 切换成本 | 建议 |
|---|---|---|---|---|
| **总体架构** | 原生 Kotlin | KMP (远期) | 中 | ✅ 保持 |
| **UI** | Compose + M3 | XML (不推荐新项目) | 高 | ✅ 保持 |
| **数据库** | Room + SQLCipher | Room + 字段级加密 | 低 | 🔶 可选 |

### 推荐的调整

基于"轻便"原则，建议对原方案做以下微调：

1. **架构简化**：去掉 UseCase 层，ViewModel 直接调 Repository
2. **加密可选**：密码本用 PBKDF2 + AES-256-GCM + 字段级加密（去掉 SQLCipher 依赖，APK 减 ~3MB）
3. **序列化切换**：考虑 kotlinx.serialization（为未来 KMP 预留）
4. **DI 保持 Hilt**：编译期安全在多模块架构中价值高
