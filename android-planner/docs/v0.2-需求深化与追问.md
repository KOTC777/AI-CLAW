## 二、需求深化分析

### 2.1 密码本多层级设计

```
┌─────────────────────────────────────┐
│  Level 0: 密码原文（最高安全级）       │  ← AES-256 加密，需主密码解锁
├─────────────────────────────────────┤
│  Level 1: 用户自定义提示              │  ← 加密存储，但可单独配置访问策略
├─────────────────────────────────────┤
│  Level 2: 待定（预留扩展）            │  ← 比如：历史密码、关联账户、备注
├─────────────────────────────────────┤
│  元数据：标题、分类、图标、创建时间     │  ← 不加密或轻加密，便于列表展示
└─────────────────────────────────────┘
```

**架构考量：**
- 每个层级独立加密/解密，支持"只看提示不看原文"的场景
- 主密码派生：PBKDF2 / Argon2 → 生成各层级密钥
- 导出时整体加密为单一 `.vault` 文件

### 2.2 强制打卡多级策略

```
强度等级（用户可选）:

Level 1 - 温和提醒
  └── 通知栏提醒，可忽略

Level 2 - 持续提醒
  └── 每隔 N 分钟重复通知，直到打卡

Level 3 - 强制弹窗
  └── 全屏 Activity 覆盖，需操作才能关闭

Level 4 - 锁屏提醒
  └── 锁屏通知 + 震动 + 铃声

Level 5 - 极端模式
  └── 前台服务持续运行，定时震动/铃声
     打卡证明验证通过才停止
```

**打卡证明验证方式：**
- 拍照（调用相机）
- 录音（指定时长）
- 屏幕截图
- 输入特定文本
- 自定义条件

### 2.3 AI 集成架构（DeepSeek）

```
用户输入灵感/笔记
       ↓
发送到 DeepSeek API（带 system prompt）
       ↓
DeepSeek 返回结构化 JSON:
{
  "category": "技术",
  "tags": ["Android", "Kotlin"],
  "summary": "...",
  "action": "organize",
  "commands": [
    {"type": "move", "target": "技术笔记"},
    {"type": "add_tag", "value": "架构设计"},
    {"type": "set_priority", "value": "high"}
  ]
}
       ↓
本地 CommandExecutor 解析并执行 commands
       ↓
更新本地数据库
```

**关键设计：**
- AI 只返回"指令"，不直接操作数据
- 本地 CommandExecutor 是唯一的数据修改入口
- 支持命令队列、回滚、日志
- 离线时缓存请求，联网后批量处理

### 2.4 网页完整存储

```
用户输入 URL
       ↓
WebView 加载页面
       ↓
资源捕获：
  ├── HTML 完整文档
  ├── CSS / JS 文件
  ├── 图片（img/video/audio）
  └── 字体文件
       ↓
资源本地化（替换 URL 为本地路径）
       ↓
打包存储：/web_snapshots/{id}/
  ├── index.html
  ├── assets/
  │   ├── images/
  │   ├── css/
  │   └── js/
  └── metadata.json
```

**技术方案：**
- `WebViewClient.shouldInterceptRequest` 拦截所有资源请求
- 或使用第三方库如 `CollapsingToolbar` 的离线能力
- 存储占用管理：提供清理策略（LRU / 按大小 / 按时间）

## 三、第二轮追问

### Q9：密码本
- a) Level 1（提示）是否需要支持多条提示？比如"这密码和XX有关"+"长度12位"
- b) 密码条目是否需要分组/分类？（如：社交、工作、金融、游戏）
- c) 导出的 `.vault` 文件是否需要支持选择性导出（只导出部分分组）？

### Q10：强制打卡
- a) 打卡任务是否需要支持"阶段式升级"？比如：截止前2小时用 Level 1，前30分钟升到 Level 3，过期后升到 Level 5
- b) 打卡证明是否需要"审核"机制？比如拍照后 AI 判断是否符合要求（如：拍的是书桌而非天空）
- c) 是否需要打卡统计/报表？（连续打卡天数、完成率等）

### Q11：AI（DeepSeek）
- a) 你是否有 DeepSeek API Key？还是需要支持多种 AI provider 切换？
- b) System Prompt 是否由你来编写，还是由我设计一套默认的？
- c) AI 处理是否需要在后台自动运行（新笔记自动分类），还是仅手动触发？

### Q12：网页存储
- a) 网页存储后是否需要支持"重新加载"（从本地打开，还原原始浏览体验）？
- b) 是否需要支持"稍后阅读"功能（类似 Pocket/Instapaper）？
- c) 存储上限是否需要设置？（防止占用过多空间）

### Q13：通用
- a) 是否需要应用锁（进入应用时需要密码/生物识别）？
- b) 数据库是否需要全库加密（SQLCipher），还是仅加密密码本模块？
- c) 是否需要支持"紧急销毁"功能（一键清除所有数据）？
